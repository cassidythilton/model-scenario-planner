Your hypothesis is testable, but it's actually two claims bolted together, and separating them is the first design decision.

The economic claim ("cost pushes demand to secondary models") is a market prediction you can't test in Bedrock — it's a forecast about buyer behavior. What Bedrock can answer is the thing that would make that forecast true or false: for which task types does a secondary model, properly configured, close the accuracy gap enough that the cost savings justify the switch? That reframing matters because "similar accuracy to frontier" as a binary is almost certainly false in the general case and almost certainly true in narrow cases. The interesting result lives in where the line falls, not whether it exists.

The confound that will sink the study if you're not careful

"Proper context and post-training configuration" is doing enormous work in your hypothesis, and if you're not disciplined it becomes unfalsifiable — any time a small model loses, you can claim it wasn't configured well enough. Two defenses against that:

First, pre-register what "match" means before you run anything. Pick a threshold (e.g., secondary model is within 3% absolute accuracy of frontier on the task) and a configuration budget (e.g., you'll allow up to N fine-tuning examples and a fixed prompt-optimization effort, then stop). Otherwise you'll keep tuning until you get the answer you already believe.

Second, and this is the control people most often forget: apply the same context engineering to the frontier model. If you build a great RAG pipeline and few-shot setup for Llama or Nova and then compare it against vanilla Claude or GPT, that's a rigged fight — the frontier model usually benefits from the same context just as much or more. The honest comparison is configured-secondary vs. configured-frontier. Your hypothesis is really "context+post-training narrows the gap faster than it helps the frontier model," and only the symmetric design tests that.

The design I'd run

A factorial: models × intervention level × task type, scored on accuracy, cost, latency, and variance.

The intervention ladder is the spine of it, run for both a secondary and a frontier model:

Base model, zero-shot
prompt optimization / few-shot
retrieval (RAG over your domain corpus)
fine-tuning (SFT, or Bedrock's reinforcement fine-tuning)
everything combined
This lets you attribute the gap-closing to specific levers, which is the actually useful output. You'll likely find context engineering (steps 2–3) closes most of the gap on retrieval/extraction/classification tasks for cents, while fine-tuning (step 4) is what closes it on narrow-domain reasoning but carries upfront cost that only amortizes at volume. Those are two very different economic stories and they feed directly back into your demand thesis.

Task taxonomy is where the result actually lives

Don't run one blended benchmark — segment by task type, because the gap behaves completely differently across them. A rough prior from what's generally observed:

The gap closes easily (secondary models win on cost with near-equal accuracy) on extraction, classification, structured-output generation, summarization, and anything RAG-friendly where the answer is in the provided context. The gap persists stubbornly on multi-step reasoning, long-horizon agentic tasks, novel/out-of-distribution problems, and tasks needing broad world knowledge the small model never absorbed in pretraining. Build your task suite to span both ends so you can map the boundary rather than landing on an average that hides it.

Critically, use your own representative task distribution if this is for a real deployment decision, not just MMLU/public benchmarks — public benchmarks are contaminated and don't reflect your workload. Bedrock's model evaluation jobs let you bring your own datasets, which is what you want.

Metrics beyond accuracy

Accuracy alone will mislead you. Track at minimum: cost per task (inference tokens, plus amortized fine-tuning cost over expected volume), latency (p50/p95), variance across runs (run each item 3–5× at non-zero temperature — small models are often less reliable, not just less accurate, and a 90%-accurate model that fails unpredictably is worse than the number suggests), and failure-mode severity (a wrong extraction vs. a confidently-hallucinated wrong answer have very different downstream costs). The deliverable should be a cost-performance Pareto frontier per task type, not a leaderboard.

Executing it in Bedrock

The tooling lines up well with this design. Bedrock's Model Evaluation supports automatic evaluation with predefined algorithms for accuracy, robustness, and toxicity, plus human evaluation and bring-your-own datasets — and it now evaluates custom fine-tuned and imported models, so you can score every rung of the intervention ladder in the same harness. For the retrieval rung, use Knowledge Bases. For fine-tuning, SFT is available across Nova, Llama, Cohere, and Titan; and notably Bedrock now has reinforcement fine-tuning, available at launch for Amazon Nova 2 Lite, which delivers an average accuracy gain of around 66% over base models and is aimed specifically at making smaller, cheaper models perform like larger ones on your specific tasks. That RFT path is almost a direct test of your hypothesis on one model family, so it's worth including as its own arm — though don't let one vendor's headline number stand in for your measurement. One practical note: Nova supports on-demand invocation of customized models so you pay per-call rather than buying provisioned throughput, whereas some other custom models require provisioned throughput, which changes your cost accounting meaningfully — fold that into the cost model.

A market-investigation framing actually makes the build simpler, because you're optimizing for breadth and repeatability of scenarios rather than depth on one deployment. What you're describing is a scenario factory, not a single benchmark — and that shapes the whole design toward a reusable harness.

Here's a draft scope to react to before I formalize it.

What the app is

A scenario-comparison harness: define a library of real-world Domo-customer tasks, run each through multiple model configurations on Bedrock, score the outputs automatically, and surface a cost-vs-accuracy picture per task type. The manual and automated modes share one engine — a "playground" for one-off side-by-side runs, and a batch runner for mass execution.

Core objects — this data model is what makes everything else fall out:

Scenario: a task prompt + optional input context + expected answer or scoring rubric + a task-type tag (extraction, reasoning, RAG-QA, summarization, classification, etc.)
ModelConfig: model + intervention level (zero-shot, few-shot, RAG, fine-tuned) + parameters
Run: one Scenario × ModelConfig execution, capturing output, tokens, latency, cost
Eval: the score(s) attached to a Run
ScenarioSet: a named collection you execute as a batch
Manual mode: pick a scenario (or type freeform), pick 2–4 configs, fire, see outputs side-by-side with scores, cost, and latency. This doubles as the surface you'd actually demo live to a customer.

Batch mode: select a ScenarioSet against a matrix of ModelConfigs, queue the runs, auto-score, persist results, render the frontier. This is where async execution and a real backend matter.

Eval engine: pluggable scorers by task type — exact/fuzzy match and structured-field accuracy for extraction and classification, LLM-as-judge against a rubric for open-ended tasks, plus always-on metrics (cost, latency, and run-to-run variance from N repeats, since reliability is half the story for smaller models).

Reporting: a cost-performance Pareto frontier per task type, a "gap-closing by intervention" view showing how much each lever buys you, and a reliability view.

Suggested MVP cuts: defer fine-tuning arms at first (zero-shot/few-shot/RAG are cheap and answer most of the market question — add SFT/RFT once the harness is proven), defer human-eval workflows, defer multi-user sharing.

Yes — and I'd actually switch my recommendation to JavaScript (ideally TypeScript). Python was only my default because boto3 is ergonomic, but the JS AWS SDK v3 (@aws-sdk/client-bedrock-runtime, with ConverseCommand / InvokeModelCommand) is fully capable — it handles SigV4 signing, usage-token reporting, and streaming. Nothing in the spec changes functionally: the normalized contract, the two adapters, and the scoring handlers are identical regardless of language.

What makes JS the better choice here, not just an equal one, is stack unification. Your frontend is already a React/JS custom app, so going JS in CodeEngine lets you:

Define the contract once — the normalized request/response shape, the registry entry type, the scorer types — as shared TypeScript types/modules used by both the React app and the function, instead of maintaining a Python copy that silently drifts.
Work in native JSON end-to-end, which is most of what this tool does (structured-field scorers especially operate directly on parsed JSON).
Stay in one mental model as a solo builder — less context-switching, fewer serialization seams.
The one thing to verify in your instance, because it's the real hinge: does CodeEngine let you pull in an arbitrary npm dependency like @aws-sdk/client-bedrock-runtime, or are you limited to its provided global-package library? Three cases:

You can add @aws-sdk → ideal. Use the SDK for the Converse path; the function is clean.
CodeEngine provides an AWS or generic HTTP package → use that.
No external deps allowed (only fetch-equivalent) → you hand-roll SigV4-signed requests against the Bedrock REST endpoints. Doable, just more boilerplate — and notably, this is the one case where JS loses its edge, since SigV4-by-hand is fiddlier in JS than boto3 is in Python. If you're equally restricted in both languages, it's a wash and JS still wins on stack unification.
Worth knowing regardless of the above: the mantle / open-weight path (Kimi, DeepSeek, Qwen) is likely a fetch call to the OpenAI-compatible Chat Completions endpoint rather than an SDK command, so that adapter is plain HTTP either way. And the reference_similarity scorer should call a Bedrock embeddings model (Titan or Cohere embed) through the same path rather than pulling in a local embedding library — so you don't need heavy ML dependencies in the function no matter which language.
